# **Backlog Report**

This report reflects on my experience building a project with Backlog.md. It covers three questions: how long the project took, where the service was genuinely strong, and where it fell short. The short version is that Backlog's onboarding got me productive almost immediately, and the project came in at 2.5 hours against a 2-hour baseline — with the overage owed to learning the CLI and tightening up the project itself, not the tool slowing me down.

## **1\. How long did the project take?**

The deliverability project took 2.5 hours to complete using Backlog, against the 2-hour baseline from building the same project on Conductor. The 30 minutes over weren't pure tool overhead: part went to getting used to the CLI, and part to tightening up the deliverability project itself along the way.

## **2\. Strengths of the service**

Backlog's strengths cluster around onboarding: the README, the CLI's task-creation flow, and the guided starter tasks all point the same direction — getting you productive immediately. The standout strengths:

* **An expansive README with a natural flow.** A quick description of what Backlog is, then immediately into creating your first task — the CLI explained in the order you'd actually use it.  
* **Rich task creation at the CLI.** Description, definition of done, acceptance criteria — many elements of a task can all be built right at CLI task creation.  
* **Onboarding tasks.** Five starter tasks designed to get you used to the service — creating your first task, launching the board, editing a task within the Kanban board. They prepared me to immediately start using the CLI.

## **3\. Weaknesses of the service**

The weaknesses show up after the start — in how the service holds up as the board fills and in how tasks can be edited. The main friction points:

* **The Kanban board doesn't scale with the task count.** With only a few tasks, the board gives the appearance of organization — but once it gets extremely filled, it becomes more overwhelming than organized.  
* **Tasks are files you can't edit as files.** Tasks are markdown files on disk, but editing the file directly isn't supported — the CLI and the Kanban board are the only edit paths.  
* **Definition of done felt redundant.** Acceptance criteria were the main thing I used to express what done meant. Backlog also offers a separate definition-of-done field, but it read as redundancy — I didn't use it once across all of the deliverability tasks.

## **Closing thought**

Backlog's best quality is how little time it takes to become productive: the README and onboarding tasks did their job, and the 2.5-hour finish — half an hour over the Conductor baseline, owed mostly to learning curve and project tightening — reflects a tool that gets out of the way early. The friction shows up after the start: a board that becomes overwhelming as it fills, tasks that live as markdown on disk but can only be edited through the CLI or the board, and a definition-of-done field that acceptance criteria already covered. Strong on the way in; the rough edges are in the day-to-day that follows.
