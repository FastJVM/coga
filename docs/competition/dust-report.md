# **Dust Report**

This report reflects on my experience building a deliverability project with Dust. It covers three questions: how long the project took, where the service was genuinely strong, and where it fell short. The short version is that Dust made the early, structural parts of the work feel effortless, but the absence of workflows turned the harder, iterative parts into a slow trial-and-error grind.

## **1\. How long did the project take?**

The project took roughly three hours to complete. My intended cutoff was two hours, and I pushed past it because I wanted to finish rather than leave the work half-done. That one-hour overrun is worth holding onto, because it points directly at where Dust helped and where it didn't: the structure came together fast, but the detailed, iterative work — the part that ran over — is exactly where the service slowed me down. For comparison, Backlog and the Linear agent were each able to finish the same deliverability project in about two hours.

## **2\. Strengths of the service**

Dust was at its best when turning intent into structure and keeping the work legible as it progressed. The standout strengths:

* **Vision-to-task translation.** I could describe the project as a step-by-step vision, and the agent turned it into a comprehensive, well-organized task list. The gap between "here's what I want to build" and "here's a concrete plan" closed almost immediately.  
* **Pods as directories.** Pods behave like directories where agents build up context. It was easy to pull in files that better explained a task, so the agent was working from real material rather than guesses.  
* **Automatic task completion.** One of the genuinely cool details: tasks auto-marked as done. I would share a message from my terminal noting that a step was finished, and Dust would recognize it and mark the task complete on its own — no manual bookkeeping.  
* **Granular, test-driven tasks.** The agent broke work down finely and walked me through testing every aspect of the project before moving on. When a batch of code didn't work, it provided next recommendations that were easy to copy and act on.  
* **A real learning experience.** Building out my own directory alongside the agent's code recommendations taught me a lot. It wasn't just output handed to me; I came away understanding the project better.

## **3\. Weaknesses of the service**

The weaknesses clustered around control and efficiency — specifically, how much had to route through the agent, and how painful iteration became without workflows. The main friction points:

* **No direct task editing.** I couldn't find a way to click into a task and edit it directly. All of that work seemed to have to go through the agent, which removed a fast, obvious path for small corrections.  
* **Painful iteration on complex work.** For something like finding the correct Playwright selectors to drive the project, the process was exhausting. I was stuck working on a trial-and-error basis — deleting and adding code repeatedly — rather than converging cleanly on a solution.  
* **Absence of workflows.** This is the root of the previous problem. Without workflows to structure repeated steps, the iterative work had no scaffolding, and the whole process became inefficient.  
* **More complexity than the task needed.** The trial-and-error process — which I think is a byproduct of having to build my own directory — made the project take far longer than it should have. Backlog and the Linear agent both finished the same deliverability project in two hours, while this approach ran to three.  
* **Imprecise capture of what was learned.** When I wanted to add things the agent had learned during the process, I had to be very specific. Otherwise it tended to batch-grab a lot of material that didn't matter, diluting what I actually wanted to keep.

## **Closing thought**

Dust excels at the front of the work — translating a vision into structure, keeping context organized in pods, and removing bookkeeping with auto-completion. Where it struggles is the long tail of iteration: without workflows and without direct task editing, complex, trial-and-error tasks become slow and exhausting. Closing that gap is what would bring the three-hour experience back down toward the two-hour mark.
