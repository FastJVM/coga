### Confirmed design — HYBRID direct-delete

Today Retro *always* deletes a processed done ticket inside a reviewable PR:
knowledge-bearing tickets in their theme knowledge PR, no-knowledge tickets via
a `## Pruned` fold or a single delete-only "prune PR". The painful/ceremonial
part is the no-knowledge case.

New behavior:

- **Knowledge-bearing ticket → UNCHANGED.** Extracted + `## Retro` marker
  recorded + directory deleted, all in its theme knowledge PR. Atomic,
  reviewable, editable, rejectable. Phase 5 (cleanup-orphan-markers) stays as
  the recovery net for that PR-gated deletion.
- **No-durable-knowledge ticket → DIRECT-DELETE immediately.** No knowledge
  edit, no `## Retro` marker, no PR. Deleted via `relay delete <slug>`
  (working-tree `git rm` + direct `Ticket: <id> — deleted` commit). Recover via
  `git restore`.

Removed concepts: delete-only prune PR, `## Pruned` section, the
`result: no-new-durable-knowledge` marker, and the marker-only-PR prohibition
(n/a for direct-deleted tickets).

Why hybrid (not full direct-delete): keeps the human's ability to
reject/edit a knowledge change *together with* its source-ticket deletion in
one atomic PR; avoids orphaning a knowledge ticket's source before its
knowledge lands. Only the ceremony around knowledge-less tickets is removed.

### Decisions (interactive Q&A with nick)

- Knowledge stays PR-gated (memory-via-PR).
- Direct deletion uses the existing `relay delete` / `bootstrap/delete-task`
  primitive (working-tree git rm + direct commit; recover via git restore).
- Keep the `## Retro` marker + Phase 5 for the knowledge case; drop them only
  for the no-knowledge case.
