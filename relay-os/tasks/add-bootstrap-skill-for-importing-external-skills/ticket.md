## Description

Add a **skill-import pass**: the active path by which Relay brings an external
Agent Skill into a repo instead of hand-writing every reusable playbook. The
pass fires at the existing gap point — when ticket authoring (`bootstrap/ticket`
step 4) finds a workflow step needs a skill that doesn't exist — and runs
search → decide (import / adapt / write) → install with provenance.

This is the *action* half of the work. The *trigger* half — detecting when a
skill is missing — is split into its sibling ticket `detect-missing-skills`.
Referenced-but-absent skills are already detected today (`relay validate`
emits `broken-skill`; `compose.py` hard-fails on a missing layer); the open
question of capability-gap detection lives in that ticket, not this one.

## Context

This came out of `add-dev-unit-test-dream-worker`, where the useful design
became "import/adapt good generic testing skills" rather than inventing a
Dream-specific worker. Prior art: OpenClaw / ClawHub — a 2026 skills
marketplace using the same `SKILL.md` format, with thousands of community
skills installable by CLI. Relay's analog is `relay skill install-url`, which
already writes `.relay-source.json` provenance; the gap is discovery (Relay has
no search command, so a candidate is found by web/GitHub browsing).

Candidate source examples found during design:

- `sanity-labs/test` - test runner shape with failure reporting
- `ingpoc/SKILLS/testing` - code-verified testing evidence
- `ArcaneArts/update-unit-tests` - low-scaffolding unit-test update workflow

Implementation gotcha (survives a fresh session): bundled bootstrap skills are
authored in the SOURCE template under
`src/relay/resources/templates/relay-os/bootstrap/skills/`, **not** the live
`relay-os/bootstrap/` copy (which is gitignored and regenerated by
`relay init --update`). The source `bootstrap/` path is itself gitignored, so
new files must be force-added with `git add -f`.

Because this is a docs-only change (two `SKILL.md` files, no code), the
`peer-review` step should check skill *content accuracy* — does the decision
tree and provenance section match the real `.relay-source.json` fields and
`relay skill` subcommands — rather than hunt for code-correctness bugs.

## Acceptance criteria

- [x] A `bootstrap/import` skill exists in the Relay skill tree, documenting
      the pass: import / adapt / write decision, registry search, provenance,
      placement, anti-patterns, worked example.
- [x] It explains when to import, adapt, or write a new skill.
- [x] Provenance reuses the existing mechanism (`relay skill install-url` +
      `.relay-source.json`: source URL, ref, install date, local-change digests,
      and hand-edited `local_adaptation_notes`) rather than a new format.
- [x] It defines where imported/adapted skills live (`relay-os/skills/<ns>/<name>/`).
- [x] It says not to blindly import broad skills that add irrelevant workflow
      or hard-coded commands.
- [x] It includes a short example import decision for a dev testing skill.
- [x] The pass is wired into `bootstrap/ticket` step 4 so an importable skill
      is checked for before one is hand-written.
- [x] Decision on pass scope: the inline `bootstrap/ticket` step-4 hook is
      sufficient for now. A richer flow (a `relay skill search`/`import`
      command, or a Dream maintenance pass that proposes imports) is **deferred**
      — not built in this ticket. Discovery stays manual (web/GitHub browsing)
      until a concrete need justifies a command.

## Out of scope

- Missing-skill *detection* — `relay validate` / `compose.py` referenced-skill
  checks and any capability-gap heuristic — is owned by the sibling ticket
  `detect-missing-skills`. This ticket adds only the import *action* plus the
  `bootstrap/ticket` step-4 hook; it does not touch `validate.py` or
  `compose.py`.
- A `relay skill search`/`import` command or Dream import pass (deferred per
  the decision above).
